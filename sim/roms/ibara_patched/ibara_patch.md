# ibara ŌĆö fast-boot simulation patch

Purpose: let the Verilator board sim reach the blitter/game code in a few hundred
instructions instead of grinding through ~1 M instructions of boot loops that are
pointless in RTL sim (we can just preload work-RAM). Everything here is opt-in via
`FASTBOOT=1` and changes **no** default-build behaviour.

Generated by `sim/scripts/make_fastboot.py` from the pristine `sim/roms/ibara/u4`.

```
FASTBOOT=1 ./build_sim.sh +maxinsn=3000000     # build patched + run
```

Artifacts:

| File | Role |
|---|---|
| `sim/roms/ibara_patched/u4_fastboot` | patched raw 2 MB NOR image (reference) |
| `sim/roms/ibara_patched/ibara_u4_4M_fastboot.hex` | patched flash `$readmemh` image (4 MB mirror) |
| `sim/roms/ibara_patched/ibara_sdram_bank0.hex` | SDRAM preload, bank 0 (P = 0x000000ŌĆō0x1FFFFC) |
| `sim/roms/ibara_patched/ibara_sdram_bank1.hex` | SDRAM preload, bank 1 (P = 0x200000ŌĆō0x3950B8) |

Wiring: `tb_cv1k.sv` selects the patched flash hex under `IBARA_FASTBOOT`;
`models/mt48lc2m32b2.v` `$readmemh`'s the two bank files at t=0 under the same
define; `build_sim.sh` passes `+define+IBARA_FASTBOOT` when `FASTBOOT=1`.

## 1. The boot loops (reset 0xA0000000 ŌåÆ WORKRAM_ENTRY ŌåÆ blitter)

The SH-3 resets into NOR flash at `0xA0000000`. After straight-line MMIO/BSC setup
(`0c000000ŌĆō0c000090`: SR/VBR/GBR, area-3 SDRAM controller, port straps) it runs:

| Loop | Addr | Iters | What it does | Sim? |
|---|---|---|---|---|
| flashŌåÆSDRAM copy | `0c000092` | ~938 k | copies `0xA0000000`ŌåÆ`0xAC000000`, `0x3950BC` bytes | **NOP'd** ŌĆö we preload SDRAM |
| 3 table patches | `0c0000a2/b2/c2` | 4/4/5 | small fix-ups into SDRAM | kept (trivial) |
| cache flush | `0c002a66` | 1024 | writes SH-3 cache addr array `0xF0000000` | kept (cheap) |
| driver clear | `0c049986` | 20 | clears blitter-driver state table | kept (trivial) |
| **FPGA upload** | `0c04a27e` | 290 k B | bit-bangs the Cyclone bitstream (see ┬¦2) | **skipped** ŌĆö `jsr` at `0c049f64` NOP'd |
| FPGA settle delay | `0c049f70` | 262 144 | `dt`/`bf` busy-wait after config | **NOP'd** (`bf/s` at `0c049f74`) |
| blitter poll | `0c049f86` | HW | spins while `[B8000010] & 0x2` (busy) is set | **kept** ŌĆö this is the "wait for blitter" |

### The three NOP patches

Each SH-3 instruction is byte-swapped in the file; NOP = `0x0009` ŌåÆ bytes `09 00`.

| CPU addr | orig | ŌåÆ | why |
|---|---|---|---|
| `0x000092,94,96,98,9a` | `6016 4310 2202 8ffb 7204` | `0009├Ś5` | copy loop; SDRAM is preloaded |
| `0x049f64` | `400b` (`jsr @r0`) | `0009` | skip `FUN_0c04a250` FPGA bitstream upload |
| `0x049f74` | `8ffc` (`bf/s`) | `0009` | drop the 0x40000-iter config settle delay |

The blitter-ready poll (`0c049f86`) is deliberately **not** patched: it loops only
while the blitter reports *busy*, so it is exactly the "move to SDRAM and wait for
the blitter" behaviour requested. (In today's sim CS6 is undecoded and reads back 0,
so the busy bit is clear and the poll falls through immediately; once the blitter
RTL drives `B8000010[1]`, this becomes a real DDR-init wait with no code change.)

## 2. Cyclone FPGA (U8, EP1C12) bitstream transmission ŌĆö **SERIAL**

`FUN_0c04a250` configures the blitter FPGA in **Passive Serial** mode by
bit-banging, **not** a parallel/FPP transfer. Details:

- Source: `0x0C002AC0` in the program image; length `0x46E65` = **290,405 bytes =
  2,323,240 bits** (read from `DAT_0c152c48`). Matches an EP1C12 `.rbf`.
- Port: SH-3 **Port J data register `0xA4000130`** (byte writes).
- Per bit (loop `0c04a27eŌĆō0c04a3c6`, bytes consumed **LSB-first**, `shlr`):
  the current data bit is placed on **Port J bit 5 (0x20) = DATA0**, and the code
  writes the byte **three** times to toggle **bit 4 (0x10) = DCLK**:
  `(DATA,DCLK=0)`, `(DATA,DCLK=1)`, `(DATA,DCLK=0)` ŌĆö i.e. one rising DCLK edge
  clocks one bit. **Bit 3 (0x08)** is held high throughout (nCONFIG/nCE-class line).
- Framing: before the loop `0xB0C00002` (U13 CPLD) toggles the FPGA config/reset
  line (asserted `=1` in `FUN_0c049f54` at `0c049f5c`, released `=0xFFFFFFFE` at
  `0c049f68`); `FUN_0c04a414` is a short `dt` delay used between phases.

So: **serial, LSB-first, one bit per DCLK pulse, DATA=PJ[5]/DCLK=PJ[4]**. In RTL the
whole thing is moot (the blitter is real logic, already "configured"), which is why
we simply skip the `jsr`.

## 3. From blitter READY to the main IRQ-wait loop

Summary of what runs after the poll at `0c049f86` passes:

| Step | Addr | Behavior |
|---|---|---|
| finish blitter init | `FUN_0c049f54` | framebuffer bases (`0x00200000`/`0x01A00000`), scroll `B8000014/18`=0, regs `B8000008/34/30/48`; heap init (`FUN_0c04a42c`); VRAM clear (`FUN_0c04a61e`) |
| subsystem inits | `WORKRAM_ENTRY` | IRQ controller (`FUN_0c04a1a8`├Ś2, `FUN_0c04a0bc` enables IRQ), YMZ770 sound (`FUN_0c0b0240`), serial RTC/EEPROM (`FUN_0c04bca8`), BSS/table clears |
| game main entry | `FUN_0c04ebc0` | read blitter DSW/status (`B8000050/00`), work-RAM warm-boot sentinel `[0x0C275DFC]==0x12345678`, NAND base `0x10000000`, display-list setup (`FUN_0c04f930`), sound+input init |
| main loop | `0c04ec70 Ōåö 0c04eda0` | infinite game-state dispatch: build sprite list ŌåÆ kick blitter ŌåÆ **wait for vblank IRQ** (VBR `0x8C000000`) |

Detail:

Once the poll at `0c049f86` passes, the remaining bring-up is:

1. **Finish blitter init** (rest of `FUN_0c049f54`): set framebuffer bases
   (`0x00200000`/`0x01A00000` in VRAM), scroll `B8000014`/`B8000018`=0, blitter regs
   `B8000008` (list/VRAM base), `B8000034/30/48`; heap allocator init
   `FUN_0c04a42c` (frees `0C3950BC..0C7FF000`); VRAM clear `FUN_0c04a61e`.
2. **Rest of WORKRAM_ENTRY** subsystem inits: interrupt controller
   (`FUN_0c04a1a8`├Ś2, `FUN_0c04a0bc` enables IRQ), `FUN_0c0b0240` (YMZ770 sound),
   **`FUN_0c04bca8` (serial RTC/EEPROM)**, table/BSS clears, then `jsr FUN_0c04ebc0`.
3. **Game main `FUN_0c04ebc0`**: reads blitter DSW/status (`B8000050`, `B8000000`),
   checks a **work-RAM warm-boot sentinel** ŌĆö `[0x0C275DFC] == 0x12345678` (`0c04ec3c`);
   if absent it runs the full init block and stamps the sentinel, if present it skips
   to steady state (soft-reset path). This is a RAM-valid marker, **not** the RTC/EEPROM.
   Then sets the NAND base `0x10000000`, calls the display-list/blitter setup
   `FUN_0c04f930` (emits blitter
   command words `C0000001h`/`F0000000h`), sound + input init, then enters the
   **infinite game-state dispatch loop** `LAB_0c04ec70 Ōåö LAB_0c04eda0`. Each pass
   builds the sprite display list, kicks the blitter, and **waits for the vblank IRQ**
   (VBR = `0x8C000000`). That is the "main infinite loop (IRQ wait loop)".

### Where the fast-boot sim currently lands (verified)

`FASTBOOT=1 ... +maxinsn=3000000`:

- reaches `WORKRAM_ENTRY` (`0c04eb20`) after **~191** boot instructions (vs ~1 M);
- FPGA upload + settle delay skipped, blitter poll passes;
- runs full game init and enters `FUN_0c04ebc0` (dispatch `0c04ec70` reached);
- finally **spins in `FUN_0c04bc34` bit-banging `0xB0C00001`** ŌĆö the serial
  **RTC/EEPROM** (U13 CPLD), which is not yet modeled (readback = 0). That is the
  next peripheral to implement; it is *past* the blitter, so the goal is met.
